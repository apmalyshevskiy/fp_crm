# FUSIONPOS Mini-CRM — резюме проекта

Документ-снимок состояния пилота на момент 14 мая 2026.
Цель документа — чтобы при возврате к работе через неделю/месяц быстро вспомнить что и как устроено.

---

## 1. Зачем эта CRM

У FUSIONPOS есть биллинг с элементами CRM. Пилот делается под **первого продавца**, чтобы:
- собирать структурированные «итоги звонков» по чек-листу из «Скрипта первичного контакта v1» (п.11)
- накапливать «золото» — дословные цитаты боли клиентов
- видеть воронку (канбан) и горящие follow-up
- анализировать причины отказов — что чинить в продукте

Принцип: каждый разговор продавца с клиентом = одна запись в `activity`, основная карточка клиента в `deals` обновляется.

## 3. Файлы на сервере и их назначение

```
/var/www/crm-api/
  ├── api.php             — PHP-CRUD-API библиотека (НЕ редактировать)
  │                         ⚠ Хвост файла (блок «// file: src/index.php» в самом конце)
  │                         был УДАЛЁН. Он автоматически выполнялся при require_once
  │                         и генерировал двойной ответ от API.
  ├── api.include.php     — наш entry point. Конфиг подключения + middlewares + require_once api.php
  └── crm.html            — фронтенд. Внутри в строках ~912-913 захардкожены API_URL и API_TOKEN

/etc/nginx/sites-available/
  └── crm-api             — конфиг Nginx (статика + проксирование в api.include.php)

/etc/nginx/sites-enabled/
  └── crm-api → /etc/nginx/sites-available/crm-api (симлинк)
```

### Важные особенности Nginx-конфига

- `location = /api.php { deny all; return 404; }` — закрыли прямой доступ к библиотеке
- `location = /api.include.php { ... }` — только наш entry point обрабатывается PHP
- Все остальные `.php` запрещены через `location ~ \.php$ { deny all; }`
- Статика отдаётся через `try_files $uri @api;` — если файл есть на диске, отдаётся он, иначе уходит в `api.include.php`

## 4. База данных

**База:** `fusionpos_crm` (utf8mb4 для русского текста и эмодзи)

**Пользователи:**
- `crm_api@'localhost'` — используется API, имеет права только на `fusionpos_crm.*`
- `root@'localhost'` — для админских операций
- `php-crud-api@'localhost'` — **БЫЛ создан** в процессе разборок, не используется, можно удалить

### Таблицы

**`deals`** — карточки клиентов, по одной строке на клиента.
Колонки сгруппированы по смыслу:
- Идентификация: `id` (VARCHAR, формата `d_xxx_yyy`), `created_at`, `updated_at`, `seller`
- Клиент: `client_name`, `phone`, `type` (формат заведения), `points`, `stage`, `open_date`, `current_system`
- Боль и потребности: `pain_quote`, `needs` (JSON-строка в TEXT), `needs_text`
- Квалификация: `temperature` (hot/warm/cold), `dm` (ЛПР), `dm_is_speaker`, `revenue`, `plan`, `hardware`
- Воронка: `status`, `next_step`, `next_date`
- Биллинг FUSIONPOS: `fp_client_id`, `fp_domain`, `fp_version` (заполняется после регистрации клиента)
- Отказы: `rejection_category`, `rejection_reason`, `rejection_reason_other`, `rejection_quote`, `can_reanimate`, `reanimate_after_date`
- Архив: `archived_at` (soft delete — карточка не удаляется, только помечается)

**`activity`** — лента касаний, N строк на одну сделку (звонок / встреча / демо / КП / счёт / заметка):
`id`, `deal_id`, `created_at`, `seller`, `type`, `summary`, `status_after`, `temperature_after`, `next_step`, `next_date`

**`pain_quotes`** — накопительная история цитат боли. Каждый звонок может добавить новую цитату:
`id`, `deal_id`, `activity_id`, `client_name`, `venue_type`, `current_system`, `quote`, `created_at`

Схема создавалась через `schema.sql` (хранится локально у автора).

## 5. Авторизация и безопасность

### Сейчас защищено
- ✅ MariaDB слушает только localhost, наружу 3306 не открыт
- ✅ Пользователь `crm_api` имеет права только на свою БД
- ✅ API-токен `X-API-Key` обязателен для каждого запроса (`apiKeyAuth.mode => required`)
- ✅ ufw: открыты только 22, 80, 443
- ✅ fail2ban защищает SSH от перебора
- ✅ Удалённый доступ к БД — через SSH-туннель (HeidiSQL / DBeaver)
- ✅ debug-режим PHP-CRUD-API ОТКЛЮЧЁН (после стадии отладки)

### Что НЕ защищено и стоит сделать
- ⚠ HTTP без HTTPS — токен и данные ходят в открытом виде. На пилоте с одним продавцом терпимо, но для прода обязательно прикрутить Let's Encrypt
- ⚠ `cors.allowedOrigins = '*'` — любой сайт может ходить в API (с токеном). После HTTPS сузить до конкретного origin
- ⚠ Токен API захардкожен в crm.html. Поскольку фронт хостится с сервера и в публичном репо его нет — пока ОК. Но при добавлении продавцов нужна нормальная авторизация по логину/паролю
- ⚠ Нет бэкапа БД — стоит настроить mysqldump по cron

### Доступы (запомните где-то надёжно)
- Root MariaDB: пароль (свой)
- Пользователь crm_api: пароль (свой)
- X-API-Key: текущий `fpos_x7Kj9mNqR3vL2pWdF8sH-GIx9CIHFXjnw`
- Root SSH: пароль (свой)

## 6. Фронтенд crm.html — как устроен

Один HTML-файл, ~2400 строк, без сборки. Внутри:

**Главные разделы (вкладки в шапке):**
- **Дашборд** — статистика: всего, горячие, тёплые, просрочены follow-up + ближайшие 5 follow-up
- **Канбан** — 6 колонок воронки (Первичный контакт → Демо назначено → Демо проведено → КП отправлено → Счёт выставлен → Оплачено) + два сворачиваемых блока справа (Отказ, Не сейчас). Drag&drop НЕТ — клик по карточке открывает детали
- **Сделки** — список с фильтрами, поиском, сортировкой, переключателем «Активные / Архив»
- **Цитаты боли** — отдельный фид pain_quotes
- **Отказы** — аналитика: реанимация (когда вернуться), по категориям, конкуренты, по конкретным причинам

**Модалка карточки сделки** — двухрежимная:
- **view** — детали сделки + кнопки «+ Новое касание», «Редактировать», «В архив/Восстановить» + история касаний снизу
- **edit** — форма по чек-листу п.11. При сохранении: апдейтит deals, добавляет новую запись в activity, при наличии pain_quote — добавляет в pain_quotes

**Блок отказа** — показывается ТОЛЬКО когда выбран статус «Отказ». Категория → подпричина (динамически зависит от категории) → цитата → чекбокс реанимации → дата реанимации.

**Конфиг — две константы в начале `<script>`:**
```js
const API_URL = '';   // пустая строка = относительные пути (тот же origin)
const API_TOKEN = 'fpos_x7Kj9mNqR3vL2pWdF8sH-GIx9CIHFXjnw';
```

## 7. Известные баги и обходы

### Баг 1: PATCH в PHP-CRUD-API падает с `Invalid parameter number`

Симптом: `PATCH /records/deals/{id}` возвращает 500 с текстом `SQLSTATE[HY093]: Invalid parameter number: number of bound variables does not match number of tokens`. Это баг в самой библиотеке (v2 main branch), не починен.

**Обход**: используем **PUT** вместо PATCH. PUT — полная замена, требует прислать все поля. Фронт сначала делает GET текущей записи, мёрджит изменения, отправляет PUT. Это даёт +1 GET-запрос на каждое обновление, но работает надёжно.

Где в коде crm.html: функция `api()`, кейсы `save` и `archive`.

### Баг 2: Двойной JSON-ответ (исправлен)

Симптом был: запросы возвращали два склеенных JSON, первый — `{"code":9999,"message":"PDOException occurred"}` от попытки PHP-CRUD-API подключиться как пользователь `php-crud-api`, второй — нормальные данные.

**Причина**: в конце `api.php` (с гитхаба) есть блок `// file: src/index.php`, который автоматически выполняется при `require_once`. Он создавал свой `Api` со своим конфигом, и наш `api.include.php` создавал второй.

**Решение**: удалили хвост `api.php` начиная со строки `// file: src/index.php`. На всякий случай в crm.html есть `parseConcatenatedJson` который берёт последний валидный JSON-объект — страховка.

### Баг 3: Пустые строки в числовых полях (исправлен)

Симптом: MariaDB ругалась `Incorrect integer value: '' for column 'points'`.

**Причина**: HTML `input type="number"` отдаёт пустую строку `''` если поле не заполнено. MariaDB в strict mode такое не принимает.

**Решение**: в функции `serializeDeal` фронта пустые строки для числовых/булевых/датных полей конвертируются в `null`.

## 8. Что входит в «save» — порядок шагов

Клик «Сохранить» в форме итога звонка → 3-4 HTTP-запроса:

1. Если сделка существующая → `GET /records/deals/{id}` (взять текущее состояние для мёрджа)
2. `PUT /records/deals/{id}` (с полным мёрджем) ИЛИ `POST /records/deals` (если новая)
3. `POST /records/activity` (новая запись в истории касаний)
4. Если `pain_quote` непустой → `POST /records/pain_quotes`
5. `GET /records/deals/{id}` (получить свежую версию для отображения в карточке)

Скорость: ~300-800 мс на полное сохранение (быстрее чем было через Apps Script).

## 9. Регулярная работа с CRM

- **Доступ продавцу:** дать ссылку `http://139.100.204.216/crm.html` и логин — токен уже в файле
- **Заполнение карточек:** продавец после каждого звонка нажимает «+ Новая сделка» или открывает существующую и жмёт «+ Новое касание»
- **Просмотр данных для управления:** руководитель смотрит дашборд, канбан и вкладку «Отказы» для аналитики
- **Прямой доступ к БД:** HeidiSQL/DBeaver через SSH-туннель — для ad-hoc запросов и экспорта в Excel

# Сколько сделок по статусам
mariadb -u root -p fusionpos_crm -e "SELECT status, COUNT(*) FROM deals WHERE archived_at IS NULL GROUP BY status;"

# Бэкап БД одной командой
mysqldump -u root -p fusionpos_crm > /root/backup_$(date +%F).sql

# Узнать заблокированные fail2ban IP
fail2ban-client status sshd
