Резюме состояния микро-проекта на текущий момент. Документ-handoff: можно
дать в новом чате, чтобы восстановить контекст без пересказа.

---

## 1. Что это

`fp-contracts` — отдельный микро-проект внутри экосистемы **FUSIONPOS CRM**.
Назначение: **конструктор договоров для партнёров** — завести партнёра и его
юрлица с реквизитами, заполнить договор по шаблону, показать, распечатать,
сохранить.

Использует **тот же бэкенд и ту же БД**, что основная CRM (отдельные таблицы).

### Зачем так, а не страницей в CRM
Фронтенд делается на **Vue 3 по всем правилам**, чтобы компоненты и логику
потом можно было переиспользовать:
- в **КП** (коммерческое предложение) — это та же задача «шаблон + переменные»;
- в **большом проекте**, фронт которого на Vue.

Логика расчётов (вознаграждение, ставки) выносится в чистый JS-модуль `lib/`,
отдельно от Vue, — чтобы подключалась куда угодно.

## 2. Стек и архитектурные решения

- **Vue 3 + Vite**, Composition API (`<script setup>`).
- **Vue Router** в hash-режиме (`#/...`) — чтобы прод-сборка работала на любом
  статическом хостинге без настройки сервера.
- **Pinia пока НЕ используется** — для нескольких экранов состояние держим в
  `ref()` внутри компонентов. Добавим, если состояние начнёт расползаться.
- Бэкенд — **PHP-CRUD-API (tqdev)**, БД — **MariaDB** (та же, что у CRM).
- Слой работы с API изолирован в `src/api/client.js` — компоненты не пишут
  `fetch` напрямую.
- **Подход к договору:** шаблон + подстановка переменных (mail-merge), НЕ
  блочный конструктор.
- **Шаблоны** будут жить в `src/lib/templates.js` как объект `{ partner: {...} }`,
  ключ = тип договора. Добавить новый тип = дописать ключ, не трогая БД.
  Сейчас планируется один тип (`partner`), но заложено под несколько.
- **id во всех новых таблицах — `INT AUTO_INCREMENT`** (осознанно сменили со
  строковых: legacy из Google Sheets тут нет, генерировать id на фронте незачем).
  Фронт НЕ передаёт id при создании — база назначает сама и возвращает.

## 3. Бэкенд и авторизация (самая нетривиальная часть)

Сервер: `http://139.100.204.216`, точка входа API — `/api.php`.

**Два уровня защиты подряд:**
1. **HTTP Basic Auth** на уровне веб-сервера (Apache, .htpasswd) — закрыта вся
   папка с API. Логин/пароль — в `.env` и `vite.config.js` (здесь не дублирую).
2. **Токен `X-API-Key`** внутри PHP-CRUD-API (middleware apiKeyAuth). Значение —
   в `.env` (`VITE_API_TOKEN`).

**Проблема, которую решали:** при кросс-доменном запросе с нестандартным
заголовком `X-API-Key` браузер шлёт preflight `OPTIONS` без авторизации, а
Basic-сервер отвечает на него `401` → запрос не проходит.

**Решение — dev-прокси Vite.** Фронт ходит на свой же `localhost:5173/api/...`,
браузер не видит кросс-домена → preflight не запускается. Vite на своей стороне
подставляет Basic-логин и переписывает путь, перенаправляя на сервер.

`vite.config.js` (ключевое):
```js
server: {
  proxy: {
    '/api': {
      target: 'http://139.100.204.216',
      changeOrigin: true,
      auth: 'ЛОГИН:ПАРОЛЬ',                 // Basic подставляет прокси
      rewrite: (path) => path.replace(/^\/api/, ''),  // срезает префикс /api
    },
  },
}
```

`.env`:
```
VITE_API_URL=/api          # через прокси (НЕ полный http://139...)
VITE_API_TOKEN=...         # токен X-API-Key
```

`client.js` шлёт заголовок `X-API-Key` (как в рабочем `crm.html`); Basic берёт
на себя прокси.

> Итоговый путь запроса: браузер → `/api/records/partners` → прокси срезает
> `/api` → `http://139.100.204.216/api.php/records/partners`.
> ВАЖНО: `VITE_API_URL` сейчас = `/api`, а `api.php` добавляется... — см.
> client.js: запросы идут как `${BASE}/records/...`. Проверено: список
> партнёров грузится. (Если что-то поедет — сверять путь в Network.)

## 4. Модель данных (3 таблицы, финальная)

Полная схема — в `schema.sql`. Связи: партнёр `1 — много` юрлиц; договор
привязан к юрлицу (и дублирует `partner_id` для фильтрации).

**partners** — партнёр как контакт
`id, created_at, updated_at, name, contact, phone, email, city, comment`

**entities** — юрлица партнёра (ИП/ООО)
`id, partner_id, created_at, updated_at, legal_name, form (ИП|ООО), inn, kpp,
ogrn (ОГРН или ОГРНИП), ogrn_date (дата регистрации), legal_address,
signer_name (имен. падеж), signer_name_gen (род. падеж, для «в лице»),
signer_role, signer_basis (Устава / свидетельства...), bank_name,
bank_account, corr_account, bik`

**contracts** — договоры
`id, partner_id, entity_id, created_at, updated_at, number, date, type,
variables (JSON — ставки и пр.), body (снимок текста шаблона), status
(draft|signed|cancelled)`

> `partners,entities,contracts` должны быть в белом списке (`tables`) в
> конфиге `api.php`.

## 5. Разбор шаблона договора (партнёрский)

По реальному файлу `Партнерский_договор_13...docx`:

**Константы (в шаблон, не в БД):** все реквизиты ФЬЮЖН СОФТ, ФИО гендиректора,
город подписания (Ростов-на-Дону), весь текст разделов 1–12, минимальные
ставки 10%/15%, срок 1 год, штраф 100 000 ₽, арбитражный суд РО.

**Переменные договора** (`contracts.variables` JSON): номер, дата, и **текущие
ставки** из п. 6.5 (привлечение % / тех. сопровождение %) — **разные у каждого
партнёра**.

**Переменные юрлица** (`entities`): legal_name+form, inn, kpp, ogrn+ogrn_date,
legal_address, банковские поля, подписант. Телефон/email — из `partners`.

**Грамматика (важно для движка merge):**
- ИП vs ООО — разные формулировки преамбулы; выбирает движок по полю `form`.
  ИП: «Индивидуальный предприниматель {ФИО}… на основании свидетельства о гос.
  регистрации ОГРНИП {ogrn} от {ogrn_date}», КПП = «—».
  ООО: «ООО "{name}", в лице {должность} {ФИО_род}, на основании {основания}».
- ФИО в родительном падеже («в лице …») хранится отдельным полем
  `signer_name_gen` (автосклонение не делаем; на будущее — либа `petrovich`).
- Короткая подпись «Фамилия И.О.» вычисляется из `signer_name` кодом.

## 6. Структура проекта и файлы

```
fp-contracts/
├── schema.sql              # схема БД (залить в ту же базу, что CRM)
├── vite.config.js          # + dev-прокси на API
├── .env                    # VITE_API_URL=/api, VITE_API_TOKEN=...
├── index.html              # сгенерён Vite
└── src/
    ├── main.js             # точка входа, подключение роутера
    ├── App.vue             # каркас: шапка + навигация + <router-view>
    ├── style.css           # тема (золотой акцент, как в CRM)
    ├── api/client.js        # обёртка над PHP-CRUD-API (db.list/get/create/...)
    ├── router/index.js      # маршруты (hash-режим)
    └── views/
        ├── PartnersView.vue   # список + создание партнёров (готово)
        └── ContractsView.vue  # заглушка
```

Команды:
```
npm run dev      # разработка (с прокси), http://localhost:5173
npm run build    # сборка в dist/ (для прода)
```
`.env` и `vite.config.js` читаются только при старте — после правок
перезапускать `npm run dev`.

## 7. Прогресс и дорожная карта

### Сделано
- **Шаг 1 — фундамент:** Vue 3 проект поднят, навигация, слой API; пробита
  двойная авторизация (Basic + токен) через dev-прокси. Список партнёров
  грузится из реальной БД.
- **Шаг 2 (начат) — партнёры:** список + форма создания со всеми полями
  (название, контакт, телефон, email, город, комментарий). id переведены на
  `INT AUTO_INCREMENT`.

### Дальше
- **Шаг 2 (продолжение):** карточка партнёра (новый роут + компонент) →
  внутри неё CRUD юрлиц со всеми реквизитами (раздел 4). Редактирование
  партнёра.
- **Шаг 3 — конструктор договоров:** `lib/templates.js` с партнёрским договором,
  размеченным `{{переменными}}`; редактор договора (заполнение переменных,
  выбор юрлица и ставок); предпросмотр; **печать через `@media print` +
  `window.print()`** (нативное «сохранить в PDF», без сторонних либ).
- **Логика вознаграждения** → в `lib/` (переиспользуемо для КП).
- **Продакшен:** сборка, CORS/один origin, секреты из фронта, смена пароля.

### Паттерн, который повторяется везде
CRUD = форма (объект `form` + `v-model`) → `db.create/update` (без id, спред
`...form`) → `load()` перечитывает список. Освоен на партнёрах, дальше тот же
на юрлицах и договорах.
