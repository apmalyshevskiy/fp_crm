# FP-CONTRACTS — контекст проекта (актуально на деплой)

Документ-handoff: можно дать в новом чате, чтобы поднять контекст без пересказа.
Статус: **рабочая система, развёрнута на боевом сервере.**

---

## 1. Что это

`fp-contracts` — микро-проект внутри экосистемы **FUSIONPOS CRM**. Назначение:
**конструктор партнёрских договоров** — завести партнёра и его юрлица с
реквизитами, собрать договор по шаблону, показать, сохранить, выгрузить в
DOCX / PDF.

Использует тот же бэкенд (PHP-CRUD-API) и ту же БД (MariaDB), что основная CRM,
но в отдельных таблицах.

### Зачем на Vue, а не страницей в CRM
Чтобы логику и компоненты можно было переиспользовать в КП (коммерческое
предложение) и в большом Vue-проекте. Логика формирования договора вынесена в
чистый `lib/` без Vue — переносится куда угодно.

---

## 2. Стек

- **Vue 3 + Vite**, Composition API (`<script setup>`), Vue Router (hash-режим).
- Pinia не используется — состояние в `ref()` внутри компонентов.
- Бэкенд **PHP-CRUD-API (tqdev)**, БД **MariaDB**. Веб-сервер — **nginx** (не Apache!).
- Библиотека **docx** (npm) для генерации Word. Печать/PDF — через `window.print()`.

---

## 3. Авторизация и путь к API (ВАЖНО — здесь было много граблей)

**Реальный путь API: `/records/...` БЕЗ `api.php` в URL.** Это выяснилось из
рабочего `crm.html`: он ходит на `http://139.100.204.216/records/<table>`, nginx
настроен на «красивый» путь. Долго били в `/api.php/records/...` — отсюда были
ошибки `1000 Route not found`.

**Авторизация API — ТОЛЬКО токен `X-API-Key`.** Никакого Basic к API не относится
(в рабочей CRM в запросах только `X-API-Key`, без `Authorization`). Basic-окно,
которое всплывало, стоит на чём-то другом, к `/records/...` отношения не имеет.

**Путь зашит в `client.js`, без зависимости от `.env`:**
```js
const BASE = import.meta.env.DEV ? '/api' : ''   // dev -> через прокси, prod -> напрямую
const TOKEN = 'fpos_...'                          // X-API-Key (как в crm.html)
```
- **dev** (`npm run dev`): запрос `/api/records/...` → Vite-прокси срезает `/api`
  → `http://139.100.204.216/records/...`
- **prod** (`npm run build`): запрос `/records/...` напрямую на тот же домен.

`import.meta.env.DEV` — встроенный флаг Vite (true только в dev). Поэтому один
`client.js` работает в обоих режимах, env-файлы для пути НЕ нужны.

> `VITE_API_URL` в `.env` / `.env.production` больше НЕ используется — можно удалить.

`vite.config.js` (dev-прокси, работает только в `npm run dev`):
```js
server: { proxy: { '/api': {
  target: 'http://139.100.204.216', changeOrigin: true,
  rewrite: (p) => p.replace(/^\/api/, ''),
} } }
```
(`auth: 'crm:...'` в прокси по факту не нужен — Basic к API не относится; можно убрать.)

## 4. Деплой (nginx, статика)

Развёрнуто на том же домене, что API: `http://139.100.204.216/contracts/`.
Тот же origin → нет CORS, нет preflight.

`vite.config.js` → `base: '/contracts/'` (путь подпапки; для корня домена было бы `/`).
Hash-роутинг → серверные rewrite не нужны.

**Цикл деплоя:**
1. `npm run build` → папка `dist/`
2. **ОЧИСТИТЬ** папку `contracts` на сервере (критично!)
3. залить содержимое `dist/` в пустую папку
4. открыть с **Ctrl+F5** (или инкогнито)

> Грабли: WinSCP не удаляет старые файлы. Каждая сборка даёт новые хэши имён
> (`index-xxxx.js`); если не чистить папку, копятся старые версии и `index.html`
> тянет устаревший скрипт. ВСЕГДА чистить целевую папку перед заливкой.

Node на сервере не нужен — там только статика + PHP-API.

## 5. Модель данных (3 таблицы, id = INT AUTO_INCREMENT)

Схема — в `schema.sql`. Партнёр `1 — много` юрлиц; договор привязан к юрлицу
(+ дублирует `partner_id`).

**partners**: id, created_at, updated_at, name, contact, phone, email, city, comment

**entities** (юрлица ИП/ООО): id, partner_id, ..., legal_name, form (ИП|ООО),
inn, kpp, ogrn (ОГРН/ОГРНИП), ogrn_date, legal_address, signer_name (имен.),
signer_name_gen (род., «в лице»), signer_role, signer_basis, bank_name,
bank_account, corr_account, bik

**contracts**: id, partner_id, entity_id, ..., number, date, type,
variables (JSON: ставки + partner_name/entity_label для списка),
body (СНИМОК — JSON-массив блоков), status (draft|signed|cancelled)

> Таблицы должны быть в белом списке (`tables`) в конфиге api.php.

## 6. Логика договора (lib/ — переиспользуемое ядро)

- **`lib/templates.js`** — константная сторона «Разработчика» (ФЬЮЖН СОФТ) +
  полный текст партнёрского договора (разделы 1–13, дословно) с `{{переменными}}`.
  Шаблоны в объекте `templates[type]`; новый тип = новый ключ.
- **`lib/merge.js`** — ЕДИНЫЙ ИСТОЧНИК: `buildBlocks(type, {partner, entity, vars})`
  → массив блоков `{t:'title'|'date'|'h'|'p'|'requisites'}`. Из блоков собираются
  и HTML (`blocksToHtml`), и DOCX. Тут же: склонения ИП/ООО (`partnerPreamble`),
  число прописью 0–99 (`numToWords`), форматы дат, короткая подпись из ФИО.
- **`lib/docx.js`** — блоки → .docx (Cambria, тело 10pt, поля 2,4/1,4/1,5 см,
  колонтитул: парафы сторон по ширине + «Стр. N из M»).
- **`lib/print.js`** — блоки→HTML → печать в чистом окне A4 (те же шрифт/поля).

**Переменные договора:** номер, дата (в `contracts`); ставки 6.5 (привлечение/
сопровождение, разные у каждого) → `variables` JSON. Минимальные ставки 10/15% —
константа в тексте.

## 7. Экраны (src/views) и потоки

- **PartnersView** — список партнёров + создание. Строки → карточка.
- **PartnerCard** (`/partners/:id`) — данные партнёра (правка/удаление) + CRUD
  юрлиц со всеми реквизитами.
- **ContractsView** (`/contracts`) — список выпущенных договоров: DOCX, Печать/PDF,
  у черновиков «Изменить» и «Подписан», удаление.
- **ContractBuilder** (`/contracts/new` и `/contracts/:id/edit`) — конструктор в
  двух режимах (создание/правка). Выбор партнёра → юрлица → номер/дата/ставки →
  живой предпросмотр. Сохраняет снимок (блоки JSON).

**Снимок и статусы:** при сохранении `body` = JSON блоков → из него и печать, и
DOCX, не пересобирая. Черновик можно редактировать (снимок пересобирается);
после «Подписан» правка закрыта (двойная защита: нет кнопки + проверка статуса
при открытии).

## 9. Что сделано / что осталось

### Сделано (всё работает в проде)
- Партнёры и юрлица: CRUD (создание, правка, удаление).
- Конструктор договоров: предпросмотр, сохранение, правка черновика, заморозка
  подписанного.
- Выгрузка DOCX (Cambria, поля, парафы, нумерация страниц) и Печать/PDF.
- Развёрнуто на боевом сервере, путь API и токен настроены.

### Возможное дальше
- Убрать неиспользуемый `VITE_API_URL` из env, `auth` из прокси.
- Сменить засветившийся Basic-пароль.
- Осиротевшие юрлица при удалении партнёра (каскад или запрет).
- Автонумерация договоров; статус «Расторгнут».
- Приблизить вид DOCX/печати к оригиналу (отступы перечислений, жирные подзаголовки).
- При желании — PDF-движок (pdfmake) как третий рендер из тех же блоков.
- Переиспользование `lib/` в КП и большом проекте (исходная цель).

### Паттерн, повторяющийся везде
CRUD = форма (объект `form` + `v-model`) → `db.create/update` (без id; спред
`...form`) → перечитать список. Один компонент в двух режимах через `editingId` /
наличие `:id` в маршруте.
