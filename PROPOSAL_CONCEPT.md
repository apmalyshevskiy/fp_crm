# FUSIONPOS Mini-CRM — концепт «Коммерческое предложение» (КП)

Документ зафиксирован 14 мая 2026. Реализация отложена — здесь только концепт,
чтобы вернуться когда дойдут руки.

---

## 1. Зачем КП в CRM

Сейчас КП делаются вручную (Word/PDF). Продавец тратит время на оформление,
формат у каждого разный, история теряется. CRM может:

- Создавать КП за минуту из готового каталога
- Хранить историю КП на сделку — видно весь торг
- Печатать в едином формате
- Авто-переводить статус сделки в «КП отправлено»

КП — это **переговорный документ**, не платёжный. Реквизитов плательщика
не нужно, бухгалтерской правды не требуется.

## 2. Принципы (зафиксированы)

| Решение | Выбор |
|---|---|
| Несколько КП на сделку | Да |
| Шапка КП | Дата, номер, комментарий |
| Редактирование КП | Мягкое: draft редактируется, sent — фиксируется (но можно править если очень надо). Лучше создавать новое КП |
| Нумерация | Просто числовой ID без префиксов: 1, 2, 3... |
| Каталог товаров | SQL-таблица `catalog_items` с самоссылающимся `parent_id` |
| Структура таблицы | Товар/услуга, кол-во, цена, ставка НДС, сумма НДС, сумма (цена и сумма ВКЛЮЧАЮТ НДС) |
| Печатная форма | HTML + браузерная печать (Ctrl+P → PDF) |
| Авто-статус сделки | КП отправлено → сделка «КП отправлено» |
| Блок КП в карточке сделки | Да |
| Копирование КП | Да |
| «От кого» | Название компании + ФИО/контакты продавца |
| Цены | Из каталога по умолчанию, продавец может править |
| Срок действия КП | По умолчанию +7 дней, можно менять |

## 3. Каталог товаров и услуг — SQL-таблица

Решено: каталог хранится в БД, не в JS. Категории и позиции в одной таблице,
самоссылающаяся иерархия через `parent_id`.

### Таблица `catalog_items`

```sql
CREATE TABLE catalog_items (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  parent_id    INT NULL,                       -- NULL = категория верхнего уровня
  name         VARCHAR(500) NOT NULL,
  unit         VARCHAR(32) NULL,               -- мес / шт / услуга / час... NULL для категорий
  price        DECIMAL(12,2) NULL,             -- NULL для категорий
  vat_rate     VARCHAR(16) NULL,               -- 'no_vat' / '5'    (пока две ставки)
  position     INT DEFAULT 0,                  -- порядок отображения
  is_active    TINYINT(1) DEFAULT 1,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_parent (parent_id),
  KEY idx_active (is_active),
  FOREIGN KEY (parent_id) REFERENCES catalog_items(id) ON DELETE CASCADE
);
```

**Признаки строки:**
- **Категория** — `parent_id IS NULL`, `unit IS NULL`, `price IS NULL`, `vat_rate IS NULL`. Просто заголовок группы для UI.
- **Позиция** — `parent_id` указывает на категорию, заполнены `unit`, `price`, `vat_rate`.

При желании можно делать многоуровневую вложенность (категория → подкатегория → товар),
но для пилота — два уровня (категория → позиция) хватит.

### Начальные данные

```sql
-- Категории
INSERT INTO catalog_items (id, name, position) VALUES
  (1, 'Лицензии',     1),
  (2, 'Услуги',        2),
  (3, 'Оборудование',  3);

-- Лицензии (parent_id=1)
INSERT INTO catalog_items (parent_id, name, unit, price, vat_rate, position) VALUES
  (1, 'Лицензия на программу для ЭВМ FUSIONPOS (реестр российского ПО №31687 от 30.12.2025)', 'мес', 3000, 'no_vat', 1);

-- Услуги (parent_id=2) — НДС 5%
INSERT INTO catalog_items (parent_id, name, unit, price, vat_rate, position) VALUES
  (2, 'Настройка кассы и запуск',                          'услуга', 15000, '5', 1),
  (2, 'Обучение персонала',                                'час',     3000, '5', 2),
  (2, 'Подключение интеграции с банком-эквайером',         'услуга',  5000, '5', 3);

-- Оборудование (parent_id=3) — без НДС
INSERT INTO catalog_items (parent_id, name, unit, price, vat_rate, position) VALUES
  (3, 'Комплект «Старт» (касса + сканер штрих-кода)',         'шт',  70000, 'no_vat', 1),
  (3, 'Комплект «Профи» (касса + сканер + ТСД + принтер)',    'шт', 150000, 'no_vat', 2);
```

### Ставки НДС в первой версии

Пока **только две** ставки:
- `no_vat` — без НДС. Применяется к лицензиям и оборудованию.
- `5` — НДС 5%. Применяется к услугам.

В будущем можно добавить 20%, 7% и другие — просто другое строковое значение в `vat_rate`.

**Правила отображения и расчёта:**
- `no_vat` — на печати в колонке «Сумма НДС» прочерк («—» или «без НДС»). В итогах НДС не учитывается.
- `5` — НДС считается «изнутри» цены: НДС = цена × 5 / 105.

### Загрузка во фронт

При открытии страницы фронт делает `GET /records/catalog_items?filter=is_active,eq,1&order=position,asc`
и строит дерево в памяти:

```js
const catalog = {
  categories: [
    { id: 1, name: 'Лицензии', items: [...] },
    { id: 2, name: 'Услуги', items: [...] },
    { id: 3, name: 'Оборудование', items: [...] }
  ]
};
```

В форме КП в выпадающем списке «Добавить из каталога» — двухуровневое меню:
сначала категория, потом конкретная позиция.

### Редактирование каталога

В пилоте — через прямой SQL или HeidiSQL. Когда станет нужно — простая
админка-страничка в CRM (отдельная вкладка для admin-роли).

**Цены при добавлении в КП заполняются из каталога, но продавец может править**
прямо в строке КП. В каталоге значение остаётся прежним.

## 4. Расчёт НДС «изнутри»

Цена в каталоге и итоговая сумма **включают** НДС. Это типично для российской
B2B-практики.

В первой версии — две ставки: «без НДС» (`no_vat`) и 5% (`5`).

### Для позиции со ставкой 5% (услуги)

Цена 15 000 ₽ за услугу, кол-во 1:
- Сумма (с НДС) = 15 000 × 1 = **15 000 ₽**
- НДС в сумме = 15 000 × 5 / 105 = **714.29 ₽** (округляем до копеек)
- Сумма без НДС = 15 000 − 714.29 = 14 285.71 ₽

### Для позиции «без НДС» (лицензии, оборудование)

Цена 3 000 ₽, кол-во 1:
- Сумма = 3 000 × 1 = **3 000 ₽**
- НДС: «—» (без НДС)
- В итогах НДС не учитывается

### Итоги КП

Пример: 12 мес лицензии (без НДС, 3000/мес) + 1 настройка (5%, 15000).

```
Позиции:
  Лицензия x 12 мес × 3000 = 36 000 ₽    [без НДС]
  Настройка × 1 × 15 000   = 15 000 ₽    [НДС 5%: 714.29 ₽]

Итого:
  Всего к оплате:           51 000 ₽
  В том числе НДС 5%:          714.29 ₽
  Без НДС:                  36 000 ₽
```

### Формулы для фронта

```js
function calcRow(item) {
  const amount = item.quantity * item.price;
  let vat_amount = 0;
  if (item.vat_rate !== 'no_vat') {
    const rate = Number(item.vat_rate); // '5' → 5
    vat_amount = amount * rate / (100 + rate);
  }
  return { amount, vat_amount };
}

function calcTotals(items) {
  let total = 0, total_vat = 0, total_no_vat = 0;
  for (const it of items) {
    const r = calcRow(it);
    total += r.amount;
    total_vat += r.vat_amount;
    if (it.vat_rate === 'no_vat') total_no_vat += r.amount;
  }
  return { total, total_vat, total_no_vat };
}
```

## 5. Жизненный цикл КП

```
[Открыть сделку] → [+ Новое КП] → создаётся draft
       ↓
   draft: можно редактировать всё
       ↓ ▼ нажать «Отправить» (или сразу «Сохранить и отправить»)
       ↓
   sent: КП на руках у клиента
       ├→ Сделка авто-переходит в статус «КП отправлено»
       ├→ Можно «Создать копию» (для торга — новая версия с другими цифрами)
       └→ Можно «Отменить» (status=cancelled, в истории остаётся)

Редактирование sent-КП:
- В UI кнопка «Редактировать» появляется, но требует подтверждения
- На печатной форме появляется пометка «Версия от ДД.ММ.ГГГГ ЧЧ:ММ»
- Лучший паттерн: «Создать копию» → правишь → отправляешь новое
```

## 6. Модель данных

### Таблица `proposals` (КП)

```sql
CREATE TABLE proposals (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  deal_id      VARCHAR(64) NOT NULL,
  date         DATE NOT NULL,
  valid_until  DATE NULL,
  comment      TEXT NULL,
  status       VARCHAR(16) DEFAULT 'draft',  -- draft / sent / cancelled
  total_amount    DECIMAL(12,2) DEFAULT 0,    -- итого с НДС
  total_vat       DECIMAL(12,2) DEFAULT 0,    -- общий НДС
  total_no_vat    DECIMAL(12,2) DEFAULT 0,    -- сумма позиций «без НДС»
  seller_id    INT NULL,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  parent_id    INT NULL,                      -- если КП — копия другого КП
  KEY idx_deal (deal_id),
  KEY idx_status (status),
  FOREIGN KEY (seller_id) REFERENCES users(id) ON DELETE SET NULL,
  FOREIGN KEY (parent_id) REFERENCES proposals(id) ON DELETE SET NULL
);
```

Номер КП = `id` (так выбрал пользователь — простая числовая нумерация).

### Таблица `proposal_items` (позиции)

```sql
CREATE TABLE proposal_items (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  proposal_id  INT NOT NULL,
  position     INT DEFAULT 0,                  -- порядок отображения
  catalog_item_id INT NULL,                    -- из какой позиции каталога (NULL для произвольных)
  name         VARCHAR(500) NOT NULL,
  unit         VARCHAR(32) NULL,
  quantity     DECIMAL(10,3) DEFAULT 1,
  price        DECIMAL(12,2) NOT NULL,         -- цена за единицу С НДС (или просто цена для «без НДС»)
  vat_rate     VARCHAR(16) DEFAULT 'no_vat',   -- 'no_vat' / '5' (пока две ставки)
  amount       DECIMAL(12,2) DEFAULT 0,        -- quantity × price
  vat_amount   DECIMAL(12,2) DEFAULT 0,        -- сумма НДС в amount
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_proposal (proposal_id),
  KEY idx_catalog (catalog_item_id),
  FOREIGN KEY (proposal_id) REFERENCES proposals(id) ON DELETE CASCADE,
  FOREIGN KEY (catalog_item_id) REFERENCES catalog_items(id) ON DELETE SET NULL
);
```

Итоги (`total_*` в proposals) считаются на фронте при сохранении и записываются
для скорости. Пересчёт по позициям не нужен — он есть в proposal_items.

## 7. UI — как продавец взаимодействует

### Карточка сделки — новый блок «Коммерческие предложения»

В детальной модалке сделки (view-режим) после блока «История касаний»
появляется блок:

```
─────────────────────────────────
Коммерческие предложения        [+ Новое КП]
─────────────────────────────────
№3   14.05.2026   15 000 ₽   sent      [📄 печать] [⎘ копия]
№2   12.05.2026   18 000 ₽   sent      [📄] [⎘]
№1   08.05.2026   25 000 ₽   cancelled [📄]
```

Клик на номер → открыть КП в режиме просмотра/редактирования.

### Форма КП

Модалка с двумя зонами:

**Шапка:**
- Номер (только для чтения, генерируется БД)
- Дата (по умолчанию — сегодня)
- Срок действия (по умолчанию — +7 дней)
- Комментарий (textarea)

**Таблица позиций:**

| # | Наименование | Ед. | Кол-во | Цена | Ставка | Сумма НДС | Всего | × |
|---|---|---|---|---|---|---|---|---|
| 1 | Лицензия FUSIONPOS… | мес | 12 | 3000 | б/НДС | — | 36 000 | ✕ |
| 2 | Настройка кассы | услуга | 1 | 15000 | 5% | 714.29 | 15 000 | ✕ |

Действия над таблицей:
- **Кнопка «+ Из каталога»** — открывает выпадающий список с категориями и позициями, клик → добавляется строка с заполненными значениями
- **Кнопка «+ Произвольная позиция»** — добавляет пустую строку, продавец вводит всё руками
- **Удалить строку** — крестик
- **Перетащить порядок** — опционально (можно отложить)

Сразу под таблицей — **итоги**:

```
Всего:               51 000 ₽
В том числе НДС 5%:     714.29 ₽
Без НДС (лицензии):  36 000 ₽
```

Снизу:
- Кнопка **Сохранить как черновик** (status=draft)
- Кнопка **Сохранить и отправить** (status=sent → сделка переходит в «КП отправлено»)
- Кнопка **Отмена**

В режиме sent-КП доступны:
- **Печать** (открывает печатную форму в новом окне)
- **Создать копию** (создаёт новое КП в draft с теми же позициями, родитель=это КП)
- **Редактировать** (с предупреждением)
- **Отменить** (status=cancelled)

## 9. Связь со статусом сделки

```
При действии «Сохранить и отправить» КП:
  1. proposals.status = 'sent'
  2. Если deal.status НЕ в ['Счёт выставлен','Оплачено','Отказ']:
     deal.status := 'КП отправлено'
  3. Добавляется запись в activity:
     type=proposal, summary='Отправлено КП №X на сумму Y'
```

Это автоматическое поведение, продавец не должен вручную двигать статус.

При создании нового КП поверх sent — статус уже «КП отправлено», ничего не меняем.
При отмене КП — статус сделки НЕ откатываем (продавец может выкатить новое КП).

## 10. План работ — поэтапно

Если/когда придём к реализации, шаги (каждый отдельным заходом):

1. **SQL** — создать таблицы `proposals`, `proposal_items`, `catalog_items`
2. **Заполнить catalog_items** — категории + начальный набор позиций (см. раздел 3)
3. **UI блока КП в карточке сделки** — список + кнопка «+ Новое»
4. **Загрузка каталога в фронт** — GET /records/catalog_items, дерево в памяти
5. **Модалка формы КП** — шапка + таблица позиций + итоги
6. **Логика расчётов** — НДС изнутри, итоги в реальном времени
7. **Логика сохранения** — POST в proposals + POST proposal_items + проставление итогов
8. **Связь со статусом сделки** — при sent → перевод
9. **Печатная форма** — отдельная HTML-страница с @media print
10. **Копирование КП** — клон позиций в новое КП
11. **Тестирование** — реальный сценарий с продавцом

Каждый шаг — 30 минут до часа работы.

## 11. Что отложили / Что обсудили

### Отложено осознанно

- **Скидки/наценки** на позицию — добавим если будет реальный кейс
- **Несколько валют** — пока ₽, других не нужно
- **Шаблоны КП** (саймпл «копировать-вставить») — копирование уже даёт это
- **Отправка по email из CRM** — пока продавец сам пересылает PDF
- **Электронная подпись** — не нужно для коммерческого предложения
- **Аналитика конверсии КП** — % КП, которые превращаются в оплату — после месяца-двух работы данных накопится, тогда нарисуем

### Обсуждено и принято решение

- ✅ Нумерация по простому ID — а не «КП-2026-001»
- ✅ Лицензия FUSIONPOS — без НДС, 3000 ₽/мес (российское ПО)
- ✅ Услуги — с НДС 5%
- ✅ Оборудование — пока без НДС
- ✅ НДС «изнутри» цены — стандартная российская практика
- ✅ Каталог в SQL (catalog_items с самоссылающимся parent_id), не в JS
- ✅ Печать через браузер — без библиотек на сервере
- ✅ Без реквизитов клиента — КП переговорный
- ✅ В «От кого» — FUSIONPOS + продавец из users

## 12. Что нужно от пользователя перед реализацией

   ```sql
   UPDATE users SET email = '...', phone = '...' WHERE login = 'crm';
   ```

3. **Возможно, добавить категории/позиции** которых не хватает в черновике
4. **Возможно, корпоративный логотип** для печатной формы (можно SVG/PNG)
5. **Подтвердить ставки НДС** — сейчас зафиксировано: лицензии и оборудование без НДС, услуги 5%
